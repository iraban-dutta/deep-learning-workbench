-------- XX -------- || -------- XX --------
# 1.3: Local Repo Exists, Remote Repo Exists
-------- XX -------- || -------- XX --------

git remote add origin <REMOTE_URL>

git remote -v

git push -u origin main

Prerequisite:
Local repository must contain at least one commit.

NOTE on -u flag:
-u flag in above command: git push -u origin main
Push to origin/main and remember this relationship for future pushes and pulls.

With -u
1st Time: git push -u origin main
2nd Time Onwards: git push OR git pull

Without -u
Every Time: git push origin main OR git pull origin main

-------- XX -------- || -------- XX --------
# 9.1: Notes: Generic 
-------- XX -------- || -------- XX --------

N1. git init creates a local repository.

N2. git clone creates a local repository and automatically links it to the remote repository.

N3. GitHub repositories must exist before code can be pushed to them.

N4. git push -u origin main sets origin/main as the default upstream branch.

N5. Always create a .gitignore before the first commit.

N6. git stash is useful when switching branches without committing unfinished work.

N7. git reset rewrites history. Use carefully.

N8. git revert is safer for shared repositories because history is preserved.


-------- XX -------- || -------- XX --------
# 9.2: Notes: Stashing
-------- XX -------- || -------- XX --------

S1. Stash temporarily saves uncommitted changes without creating a commit.

Example:

Current branch: feature-transformer
Modified files:
attention.py
encoder.py

Need to urgently switch to main branch and fix a bug.

Without stash:
Git may prevent branch switch due to uncommitted changes.

Use:
git stash
git checkout main

After bug fix:
git checkout feature-transformer
git stash pop

S2. git stash apply vs git stash pop

git stash apply
-> Restores stash
-> Keeps stash entry

git stash pop
-> Restores stash
-> Removes stash entry

Example:

git stash push -m "half-complete attention layer"
git stash apply

Result:
Changes restored
Stash still available

git stash pop

Result:
Changes restored
Stash deleted

Note:
If conflicts occur during git stash pop,
the stash entry may be retained until conflicts are resolved.

S3. Use stash for:
- Branch switching
- Quick experiments
- Temporary work-in-progress

Avoid stash for:
- Long-term storage
- Important backups

Create commits instead.

S4. View stash contents

git stash show

Detailed diff:
git stash show -p stash@{0}


-------- XX -------- || -------- XX --------
# 9.3: Notes: Undo Operations
-------- XX -------- || -------- XX --------

U1. git restore --staged file.py

Purpose:
Remove file from staging area.

Example:
git add model.py
Oops.
git restore --staged model.py

Result:
model.py remains modified
model.py is no longer staged

U2. git restore file.py

Purpose:
Discard uncommitted local changes.

Example:

Modified:
model.py

Want original version from last commit.
git restore model.py

Result:
Local changes since the last commit are lost.

U3. git reset --soft HEAD~1

Purpose:
Undo last commit but keep all code changes.

Example:

git commit -m "wrong message"
git reset --soft HEAD~1

Result:
Commit removed
Changes remain staged

Useful when:
- Wrong commit message
- Forgot files in commit

U4. git reset --hard HEAD~1

Purpose:
Undo last commit and delete all associated changes.

Example:

git reset --hard HEAD~1

Result:
Commit removed
Code changes removed

WARNING:
Data loss possible.

U5. git reset --hard <COMMIT_HASH>

Purpose:
Move repository back to an older commit.

Example:

History:
A -> B -> C -> D

Current:
D

Run:
Assume B has hash abc123
git reset --hard abc123

Result:
A -> B

Commits C and D disappear from current branch.

WARNING:
Use carefully on shared branches.

U6. git revert <COMMIT_HASH>

Purpose:
Safely undo an older commit.

Example:

History:
A -> B -> C -> D

Need to undo C.
Run:

Assume C has hash xyz789
git revert xyz789

Result:
A -> B -> C -> D -> E

Where:
E reverses changes introduced by C.

Advantage:
History preserved.
Safe for shared repositories.

U7. Difference: reset vs revert

git reset
Rewrites history
Removes commits
Dangerous on shared branches

git revert
Preserves history
Adds new commit
Safe on shared branches


U8. git reflog

Purpose:
View historical HEAD positions.

Useful when:
- Accidentally used git reset --hard
- Lost a commit
- Need to recover previous state

Example:

git reflog

Output:
abc123 HEAD@{0}: reset: moving to HEAD~1
def456 HEAD@{1}: commit: Added attention layer

Recover:

git reset --hard def456

Result:
Lost commit restored.

Note:
git reflog is local to your machine and can often recover commits that no longer appear in git log.
